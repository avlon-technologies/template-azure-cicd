# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this repo is

A CI/CD pipeline demo: a minimal .NET 10 web API (`cicd-demo.sln` — `Api` is a minimal-API project with a hello-world endpoint at `/`; `Api.Test` holds xUnit integration tests using `WebApplicationFactory`) deployed to Azure App Service via GitHub Actions. `Api/Program.cs` ends with `public partial class Program { }` to make the entry point visible to the test project — keep it when editing.

## Build and test commands (as run by CI)

```sh
dotnet restore cicd-demo.sln --locked-mode
dotnet build cicd-demo.sln --configuration Release --no-restore
dotnet test Api.Test/Api.Test.csproj --configuration Release --no-build --logger trx --collect:"XPlat Code Coverage"
dotnet publish Api/Api.csproj --configuration Release --no-build --output ./publish
```

Dependencies are locked: `packages.lock.json` per project (generated by `RestorePackagesWithLockFile` in `Directory.Build.props`), restored with `--locked-mode` in CI so unreviewed transitive changes fail the build. After changing a `PackageReference`, run `dotnet restore` and commit the updated lock files. The SDK is pinned by `global.json` (10.0.1xx, `rollForward: latestFeature`).

Run locally: `dotnet run --project Api`

## Workflow architecture

Three reusable workflows (prefixed `_`) are composed by branch-triggered entry workflows:

- `_build.yml` — build, test, publish, upload `webapp-publish` artifact. Accepts an optional `version` input; otherwise the build label defaults to `YYYYMMDD.<run_number>`. The label is stamped into `InformationalVersion` at the *build* step (publish runs `--no-build`) and surfaces in Swagger. Outputs `build-label`. Commit-keyed (promotable) builds also upload a marker artifact `<artifact-name>.label.<label>` whose *name* records the stamped label — the stg pipelines' reuse check reads it without downloading anything. Requires `checks: write` and `pull-requests: write` on the caller to publish test results as a PR check via `EnricoMi/publish-unit-test-result-action`. Every published artifact carries a `SHA256SUMS` manifest; with `attest: true` (set by all deployable builds — not PR builds, since fork PRs can't be granted `id-token: write`, which is also why `_build.yml` declares no job permissions and inherits the caller's grant) the manifest gets a GitHub build-provenance attestation and an SBOM companion artifact (`<artifact-name>.sbom.spdx.json`). Callers setting `attest: true` must grant `id-token: write` + `attestations: write`.
- `_image.yml` — packages the tested publish artifact as a container image via `az acr build` (runs in ACR Tasks — no Docker on any runner) using the repo-root `Dockerfile` (runtime-only, `COPY publish/` — never a source rebuild, so the tested bytes are the shipped bytes). Tags `<image-repository>:sha-<sha>` and `:<label>` in the shared registry (`ACR_NAME` repo variable), resolves and outputs the immutable **`image-digest`**, signs a build-provenance attestation on the digest, and for commit-keyed builds uploads a digest marker artifact `<artifact-name>.image.<digest-hex>` (the promotion paths read digests from these markers on provenance-verified runs — never from mutable tags). The registry repository is the **required** `image-repository` input (no default); every entry workflow threads it from the `IMAGE_REPOSITORY` repo variable (`${{ vars.IMAGE_REPOSITORY }}`), so a reuser retargets the repository by setting one variable and `_image.yml`/`_deploy.yml` stay in agreement. An unset/empty variable **fails hard** — `_image.yml`'s "Build and push image" and `_deploy.yml`'s "Resolve deployment target" both guard against an empty repository rather than shipping an unqualified image name. Runs under a GitHub environment for OIDC: dev builds use dev, release/hotfix/support use stg, the prod rebuild escape hatch uses prod (so the prod approval gate covers the image build too). Callers grant `id-token: write`, `contents: read`, `attestations: write`.
- `_deploy.yml` — logs into Azure via OIDC, **cryptographically verifies the image attestation** (`gh attestation verify` against the digest, pinned to this repo's `_image.yml` as signer; registry auth via an ACR token from the deploy identity's AcrPull) and deploys the **digest-pinned image** (`<image-repository>@sha256:…`, the `image-repository` input matching what `_image.yml` built and attested; the digest shape is validated and tags are never deployed) to the environment's Container App (`CONTAINERAPP_NAME`, `RESOURCE_GROUP` environment variables). With `traffic-shift: true` (prod — the app runs `Multiple` revision mode): pin traffic to the current revision → stage the image as a **zero-traffic revision** (suffix = sanitized label; an existing same-label revision is reused only when its image matches, otherwise hard error) → smoke test the revision's own FQDN → shift 100% traffic → post-shift smoke test through the gateway; a failed *post-shift* test automatically shifts traffic back to the previous revision. dev/stg (`Single` mode) just update in place. Smoke tests assert `/v1/hello` *and* that `/openapi/v1.json` reports the deployed `build-label` (a pre-release of it also passes — promoted images keep their candidate stamp), with 30s curl timeouts × 15 attempts to absorb scale-to-zero cold starts; the URL is `GATEWAY_URL` when the environment defines it, else the app FQDN. Prints the summary card, and with `tag-deployment: true` (stg/prod callers) tags `build/<environment>/<build-label>` on the commit the image was built from (`source-sha` input; tags are immutable — an existing tag at a different commit fails the deploy). Runs on the self-hosted runner (`runner-labels` input, default `["self-hosted"]`; needs `az`, `gh`, `curl`, `jq` — github-runner image ≥ 1.4.0) because the app's single ingress is IP-allowlisted. Callers grant `id-token: write`, `contents: write`. Failure or cancellation notifies `DEPLOY_ALERT_WEBHOOK` when set.
- `on-pr.yml` — builds/tests PRs to `develop`/`main` (required check), runs the workflow-script test suites (`.github/scripts/*.test.sh` — extraction-based: `test-lib.sh` pulls step `run:` blocks from the workflow YAML at test time, covering version guards, smoke loops, provenance/marker parsing, deploy staging/traffic branches, the attestation-verify step, the stg/prod artifact reuse and promotion gates, the reconcile audit, and the backmerge git flows), lints all workflows with actionlint + shellcheck (pinned, checksum-verified), and verifies `dotnet format`; rulesets also enforce PR-only merges (merge-commit-only on `main`). `codeql.yml` runs C# static analysis (PRs, develop pushes, weekly). `reconcile-releases.yml` (scheduled) audits recent `main` merges for versions with neither a `build/prod/*` tag nor a GitHub Release — the signature of a prod run superseded while queued — and fails/alerts with recovery commands.

Pipeline behavior also depends on repo settings outside these files: rulesets, workflow permissions (default token read-only; "Actions can create PRs" enabled for the back-merge step), the `AZURE_TENANT_ID`/`AZURE_SUBSCRIPTION_ID`/`ACR_NAME` repo variables, the per-environment `AZURE_CLIENT_ID`/`CONTAINERAPP_NAME`/`RESOURCE_GROUP` variables (optional per env: `GATEWAY_URL` for gateway-fronted smoke tests, `DEPLOY_ALERT_WEBHOOK` for failure notifications), a registered **self-hosted deploy runner** (deploys queue forever without one), and required reviewers + `main`-only branch policy on the prod environment — see the "Repository settings" section of `docs/operations-manual.md`. Gotcha: jobs downstream of the promotion path's skipped `build` job need explicit `!failure() && !cancelled()` conditions (implicit `success()` fails on skipped ancestors).

Entry workflows carry `concurrency` groups so deploys to the same environment serialize (`deploy-dev` / `deploy-stg` / `deploy-prod`; the three stg pipelines share a group on dispatch, group per-branch on push); `on-pr.yml` cancels superseded PR builds. All actions are pinned to commit SHAs with a `# vX.Y.Z` comment — Dependabot (`.github/dependabot.yml`) keeps the pins current; when bumping manually, update the SHA and the comment together.

Branch → environment mapping:

| Entry workflow | Trigger branches | Environment / Container App |
|---|---|---|
| `on-develop.yml` | `develop` | dev / `ca-cicd-demo-api-dev-cc` |
| `on-release.yml` | `release/**`, `milestone/**` | stg / `ca-cicd-demo-api-stg-cc` |
| `on-hotfix.yml` | `hotfix/**` | stg / `ca-cicd-demo-api-stg-cc` |
| `on-support.yml` | `support/**` | stg / `ca-cicd-demo-api-stg-cc` |
| `on-main.yml` | `main` | prod / `ca-cicd-demo-api-prod-cc` |

Release specifics (`on-release.yml`): pushes only build (deploy is gated on `workflow_dispatch`; no dispatch inputs). Labels are **commit-derived** — a pure function of branch name + commit height (`git rev-list --count HEAD`, hence the prepare job's `fetch-depth: 0` checkout): `release/X.Y.Z` at height N builds `X.Y.Z-build.N`, `milestone/<name>` builds `ms-<name>.N`. Because every build of a commit yields the same label, *push* builds are already promotable: they upload commit-keyed artifacts (`webapp-<sha>` + a `.label.<label>` marker + a `.image.<digest-hex>` marker from the image job, 90-day retention), and a dispatch **reuses the push artifact and skips the build entirely** after verifying its provenance — the artifact's producing run must be this pipeline building this exact commit, since artifact names are forgeable by any workflow in the repo; the shared check lives in `.github/scripts/find-verified-artifact.sh` (tested on every PR by `on-pr.yml`'s "Test workflow scripts" job, trusted-workflow lists hardcoded per caller — keep them in sync with workflow renames) (a missing/expired/unverifiable artifact rebuilds; a marker that disagrees with the derived label — possible only if the derivation logic changed between push and dispatch — deploys under the *stamped* label, since the version is compiled in and re-labeling would ship a binary that reports the wrong version). When the release merges to main, `on-main.yml` **promotes** that exact candidate (build and image jobs skipped) after the same provenance verification (producing run must be a release/hotfix/support pipeline building the release head — artifacts that merely carry the right name are rejected, and prod **fails hard** rather than falling back), deploying the image digest recorded by the candidate's digest marker (a candidate without one — built before container deploys — also fails hard) — a missing/expired artifact, or a release head with no `build/stg/*` tag (built but never green on staging), **fails the prod run** (no silent rebuild of untested source; re-dispatch the release pipeline to stage a fresh candidate, verify staging, then re-run). Labels are charset-restricted (letters/digits/`.`/`-`) because they become artifact names and git tags — a branch name that derives an invalid label fails the build with a rename instruction.

`on-main.yml` accepts an optional `version` dispatch input for rollback/redeploy: supply a previously released version (e.g. `1.5.0`) to promote that release's stg-tested artifact to prod via the same blue/green path. Requires the artifact to be within its 90-day retention window. The `release` and `backmerge` jobs are skipped on dispatch (push-only). Dispatching without a version redeploys the current HEAD of `main`. The `backmerge` job runs **parallel to deploy** (needs only `prepare`) so develop gets the merged commits back even if the prod deploy fails or waits on an environment approval: release sources get a branch back-merge PR; hotfix/support sources get a cherry-pick (`-m 1` of the merge commit) onto a `backport/<version>` branch instead, because those branches are cut from `main` or old version tags — their history is foreign to develop.

`on-develop.yml` accepts an optional `build-label` dispatch input: supply a previous dev build label to rebuild HEAD stamped with that label and redeploy to dev (useful for retrying a failed deploy with the same label). Dev deploys are not tagged — the label is validated against the workflow's own run history (labels are `YYYYMMDD.<run_number>`, so the original run is findable); dispatching without a label behaves identically to a push.

Hotfix/support specifics (`on-hotfix.yml`, `on-support.yml`): same dispatch-gated pattern as releases — pushes build and test with the commit-derived label (`X.Y.Z-hotfix.<height>` / `X.Y.Z-patch.<height>`) and upload promotable commit-keyed artifacts; dispatch reuses the push artifact and deploys to stg (no dispatch inputs). Merging either to `main` auto-detects the version from the merge commit subject (regex covers `release|hotfix|support`; a push whose subject doesn't parse **fails the run** rather than falling back to an untested date-label rebuild — only a version-less manual dispatch may rebuild `main` for prod) and triggers the same prod promotion and GitHub Release creation as a release merge; both get a cherry-pick backport PR to develop instead of a branch back-merge (see above). Cut `hotfix/X.Y.Z` and `support/X.Y.Z` branches from `main` (or a version tag), not `develop`.

## Infrastructure

Lives in the platform Terraform repo: `../infrastructure` (`avlon-technologies/infrastructure`) — module `infra/modules/cicd-demo/`, environment roots `infra/environments/cicd-demo/{dev,stg,prod}/` (hosted there rather than here per its ADR-0002; naming and layout follow `../engineering-standards`). Per its ADR-0008 (scale-to-zero Container Apps, replacing the former App Services), each environment's resource group `rg-cicd-demo-<env>-cc` contains a Consumption-only Container Apps environment `cae-cicd-demo-<env>-cc` with a Log Analytics workspace `log-…`, the API as a scale-to-zero Container App **`ca-cicd-demo-api-<env>-cc`** (0.25 vCPU / 0.5 Gi, `min_replicas = 0`, ingress target port 8080 — the name the deploy workflows target via `CONTAINERAPP_NAME`; prod runs `Multiple` revision mode for the pipeline's blue/green traffic shifts), the app's pull identity `mi-cicd-demo-<env>-cc` (AcrPull), and the deploy identity below. Images live in the shared registry `acrplatformsharedcc` (`ACR_NAME` repo variable, repository `cicd-demo/api` — set via the required `IMAGE_REPOSITORY` repo variable); Terraform seeds a sample image and ignores image/traffic drift — **the pipeline owns both**. The app's single ingress is IP-allowlisted (gateway + deploy CIDRs), so deploy jobs run on a **self-hosted runner** whose egress IP is allowlisted; each environment's `GATEWAY_URL` variable points the post-deploy smoke test at the gateway (fallback: the app FQDN, queried at deploy time). Cold starts after idle are expected (seconds to tens of seconds) — the smoke-test loops absorb them. Remote state: `sttfstatesharedcc` / `tfstate-<env>` / `cicd-demo.tfstate`.

## Deploy identity

CI deploys log into Azure via OIDC using one user-assigned managed identity per environment (`mi-github-cicd-demo-<env>-cc`, in the workload's own resource group), each with a federated credential for its GitHub environment (`repo:avlon-technologies/template-azure-cicd:environment:<env>`), **Container Apps Contributor** scoped to its own resource group, and **Container Registry Tasks Contributor + AcrPull** on the shared registry (`az acr build` and digest/metadata reads). Client IDs live in each GitHub environment's `AZURE_CLIENT_ID` variable (read by `_image.yml` and `_deploy.yml`): dev `66b2a296-…`, stg `964871d3-…`, prod `a7cf94d6-…`. Deploys fail until the Terraform in `../infrastructure` has been applied for the target environment (the Container App must exist).
